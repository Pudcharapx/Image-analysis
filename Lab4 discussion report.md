## สัปดาห์ที่ 1: การติดตั้งเครื่องมือและการจัดการไฟล์ภาพเบื้องต้น

จากผลรันจริงในไฟล์ `Lap_1_66114540443_พัชรพล_ยินดีรมย์.pdf` (ภาพทดสอบ: `Dog.jpg`):

### โจทย์ข้อที่ 1: Manual Grayscale (Y = 0.299R + 0.587G + 0.114B) เทียบกับ `cv2.cvtColor`

**คำตอบ:** "ไม่เห็นความแตกต่างระดับสายตา แต่ละระดับเชิงค่าตัวเลขมีต่างกันเล็กน้อย +- ไม่เกิน 1-2 พิกเซล"

ความคลาดเคลื่อน ±1-2 ระดับสีเทานี้ ผมมองว่าเกิดจากการปัดเศษ (rounding) ที่ต่างกันระหว่างการคำนวณลอยตัว (float) เองด้วย `np.clip(...).astype(np.uint8)` กับวิธีการปัดเศษภายในของ `cv2.cvtColor` ที่อาจใช้ fixed-point arithmetic หรือสัมประสิทธิ์ที่ปัดละเอียดกว่าเล็กน้อย เมื่อเทียบกับช่วงค่าทั้งหมด 0-255 แล้วค่าคลาดเคลื่อนระดับนี้ถือว่าน้อยมาก (น้อยกว่า 1%) จึงไม่ส่งผลต่อการมองเห็นด้วยตาเปล่า แต่สรุปได้ว่าสูตร manual กับฟังก์ชันสำเร็จรูปของ OpenCV จะไม่ให้ผลลัพธ์ที่ตรงกัน 100% เสมอไป หากงานใดต้องการความแม่นยำระดับพิกเซล (pixel-exact) ก็ต้องตระหนักถึงความคลาดเคลื่อนนี้ไว้ด้วย

### โจทย์ข้อที่ 2: ผลกระทบของการสลับ BGR ↔ RGB ต่องานสกัดวัตถุด้วยสี

**ผลการรัน:** ภาพขนาด 1400×1815 พิกเซล, ROI ที่สกัดตรงกลางภาพขนาด 200×200 พิกเซล ที่พิกัด (600, 807) → (800, 1007)

**คำตอบ:** "การสับเปลี่ยนสัญกรณ์แชนเนลสีจาก BGR เป็น RGB ส่งผลกระทบต่องานสกัดวัตถุที่มีสีเฉพาะเจาะจงอย่างมีนัยสำคัญ เนื่องจาก OpenCV อ่านภาพในรูปแบบ BGR ตามค่าเริ่มต้น หากผู้ใช้อ้างอิง index แชนเนลโดยไม่แปลงรูปแบบก่อน จะทำให้การ threshold ผิดแชนเนล"

ยกตัวอย่างให้เห็นภาพชัดขึ้น: ถ้าผมต้องการ threshold หาวัตถุสีแดง แต่ลืมแปลง BGR→RGB ก่อน แล้วไปอ้างอิง index [0] คิดว่าเป็น Red ทั้งที่จริงคือ Blue การ mask ที่ได้จะดึงวัตถุสีน้ำเงินออกมาแทนที่จะเป็นสีแดงตามที่ตั้งใจ งานที่ใช้ HSV color space (ซึ่งมักแปลงต่อจาก BGR ด้วย `cv2.COLOR_BGR2HSV`) ก็มีความเสี่ยงแบบเดียวกัน หากสลับลำดับอินพุตผิดตั้งแต่ต้น ค่า Hue ที่คำนวณได้จะคลาดเคลื่อนไปจากที่ควรจะเป็นทั้งหมด ดังนั้นจึงต้องตรวจสอบและแปลงพื้นที่สีให้ถูกต้องก่อนเริ่มงานสกัดวัตถุด้วยสีทุกครั้ง

## สัปดาห์ที่ 2: Pixel Access, ROI, Channel Splitting และ Thresholding

จากผลรัน:

```
โครงสร้างของอาเรย์: (911, 1366, 3)
ชนิดข้อมูลพิกเซล: uint8
ค่าสี ณ พิกัด (100, 150): [0, 74, 50]   <- ลำดับ BGR
```

ค่าพิกเซล `[0, 74, 50]` ที่ตำแหน่ง (100,150) อยู่ในลำดับ BGR (Blue=0, Green=74, Red=50) ซึ่งหมายความว่าจุดนี้มีโทนสีเขียวอมเหลืองอ่อนๆ (ไม่มีน้ำเงินเลย เขียวมากกว่าแดงเล็กน้อย) ผมต้องระวังไม่นำค่านี้ไปตีความแบบ RGB โดยตรง มิเช่นนั้นจะเข้าใจสีผิดไปเลย

การเขียนแทนค่าสีลงบน slice ของอาร์เรย์ (เช่น `img[50:100, 100:300] = [0,255,0]`) เป็นการแก้ไขค่าพิกเซลโดยตรงแบบ in-place ซึ่งเร็วกว่าการ loop ทีละพิกเซลมาก เพราะ NumPy ใช้ vectorized broadcasting ภายใน เช่นเดียวกับการแยกช่องสีด้วย `img[:, :, 2]` (array slicing) ที่เร็วกว่า `cv2.split()` เพราะเป็นการสร้าง view/copy ของอาร์เรย์แบบตรงไปตรงมาโดยไม่ต้องผ่าน function call overhead ของ OpenCV

ส่วนการทำ Thresholding แบบ manual (`img_binary[img_gray < 128] = 0`) ผมพบว่าให้ผลลัพธ์เทียบเท่ากับฟังก์ชันสำเร็จรูป `cv2.threshold()` เพราะใช้หลักการเดียวกันคือ fixed threshold ที่ 128 เพียงแต่เขียนด้วย boolean mask indexing ของ NumPy เอง สรุปได้ว่าการเข้าใจโครงสร้างอาร์เรย์และการใช้ slicing/masking ของ NumPy อย่างถูกต้อง ทำให้เขียนโค้ดที่ทำงานได้ผลเทียบเท่าฟังก์ชันสำเร็จรูป โดยไม่ต้องพึ่งพา loop เลย

## สัปดาห์ที่ 3: Gamma Correction, Histogram/CDF, Contrast Stretching และ Histogram Equalization

จากผลรันจริงในไฟล์ `LAB_3.ipynb`:

```
ขนาดรูปภาพนำเข้า: (911, 1366) | ชนิดข้อมูล: uint8
ค่าพิกเซลต่ำสุด (r_min) = 0
ค่าพิกเซลสูงสุด (r_max) = 255
ผลลัพธ์จาก Manual ตรงกับ OpenCV 100% หรือไม่?: False
ค่าความคลาดเคลื่อนเฉลี่ย (MAE) = 0.5167 พิกเซล
```

**Contrast Stretching ไม่มีผลกับภาพนี้เลย** เพราะ `r_min = 0` และ `r_max = 255` อยู่แล้วตั้งแต่ต้น (ภาพมีค่าพิกเซลครอบคลุมเต็มช่วง 0-255 อยู่แล้ว) สูตร `s = ((r - r_min)/(r_max - r_min)) * 255` จึงคำนวณแล้วได้ค่าเท่าเดิมทุกพิกเซล ภาพก่อนและหลัง stretch จึงเหมือนกันทุกประการ ผมสรุปได้ว่าภาพต้นฉบับ (F1.png) ที่ใช้ทดสอบไม่ใช่ภาพ low-contrast จริงๆ ตามที่ใบงานตั้งใจให้ทดสอบ (`low_contrast.png`) การจะเห็นผลของเทคนิคนี้ชัดเจนต้องใช้ภาพที่มีคอนทราสต์ต่ำจริงๆ

**ทำไม Manual Histogram Equalization ไม่ตรงกับ `cv2.equalizeHist()` 100% (MAE = 0.5167):** ทั้งสองวิธีใช้สูตรทางคณิตศาสตร์เดียวกันคือ `s_k = round((L-1) × CDF(r_k))` แต่ค่าคลาดเคลื่อนเฉลี่ยประมาณ 0.5 พิกเซลที่เกิดขึ้นน่าจะมาจากวิธีปัดเศษที่ต่างกัน — `np.round()` ปัดแบบ "round half to even" (banker's rounding) ในขณะที่ OpenCV ภายในอาจใช้การปัดเศษแบบ C/C++ ที่ปัดขึ้นเสมอสำหรับ .5 (round half up) ทำให้บางค่าต่างกัน ±1 ระดับสีเทา ค่าความคลาดเคลื่อนเฉลี่ย 0.5167 ถือว่าน้อยมาก (น้อยกว่า 1 ระดับความสว่างจาก 256 ระดับ) จึงมองด้วยตาเปล่าแทบไม่เห็นความต่าง แต่ในเชิงตัวเลขไม่ถือว่า "เหมือนกันทุกบิต" ผมสรุปได้ว่าความแตกต่างนี้เป็นเรื่องปกติของการเปรียบเทียบ manual implementation กับ optimized library function ที่มีรายละเอียดการปัดเศษภายในต่างกันเล็กน้อย ไม่ได้แปลว่าสูตรที่เขียนเองผิด

**Gamma Correction:** ค่า Gamma < 1 (เช่น 0.4) ทำให้ภาพสว่างขึ้น เพราะการยกกำลังด้วยเลขน้อยกว่า 1 กับค่าที่ normalize แล้ว (0-1) จะให้ผลลัพธ์สูงขึ้นกว่าค่าตั้งต้น ส่วน Gamma > 1 (เช่น 2.5) ทำให้ภาพมืดลงด้วยหลักการเดียวกันในทิศทางกลับกัน ผมมองว่าเทคนิคนี้เหมาะกับการแก้ไขภาพที่ถ่ายในสภาพแสงไม่เหมาะสม (มืดเกินไปหรือสว่างเกินไป) โดยไม่ต้องเปลี่ยนช่วงค่าพิกเซลทั้งหมดแบบ linear เหมือน contrast stretching

### 1. การอภิปรายผลเชิงตัวเลขและความเร็ว (Execution Time Analysis)

จากตารางเวลาข้างต้น ความต่างเกิดจาก **สถาปัตยกรรมการทำงานที่ต่างกันโดยพื้นฐาน** ระหว่างโค้ด Python ล้วนกับโค้ด C ที่ compile ไว้แล้ว ผมขออธิบายเป็นชั้นๆ ดังนี้:

- **Dynamic typing ของ Python:** ทุกครั้งที่ loop เข้าถึงตัวแปร (เช่น `window[m, n]`) Python ต้องตรวจสอบชนิดข้อมูลของตัวแปรนั้นใหม่ทุกครั้ง (เพราะตัวแปรใน Python ไม่ได้ผูกชนิดข้อมูลตายตัวตั้งแต่ compile time) ซึ่งเป็นภาระงานที่เพิ่มเข้ามาในทุกๆ การเข้าถึงพิกเซล ต่างจากภาษา C ที่รู้ชนิดข้อมูล (เช่น `unsigned char`) ตายตัวตั้งแต่ compile time จึงไม่ต้องเสียเวลาตรวจสอบซ้ำ
- **Interpreter overhead:** โค้ด Python ไม่ได้ถูกแปลงเป็นภาษาเครื่อง (machine code) ไว้ล่วงหน้า แต่ต้องผ่านตัวตีความ (interpreter) แปลทีละบรรทัดขณะรัน (runtime) ทุกรอบของ loop ยิ่ง loop ซ้อนกันหลายชั้น (แถว × คอลัมน์ × K × K) ยิ่งมีจำนวนรอบที่ต้องตีความมหาศาล ในขณะที่ `cv2.blur()` ถูก compile เป็น machine code (native binary) ไว้ล่วงหน้าแล้ว จึงรันตรงบน CPU ได้ทันทีโดยไม่ต้องตีความอะไรเพิ่ม
- **C-compiled optimized vectorization:** นอกจากจะ compile ไว้ล่วงหน้าแล้ว OpenCV ยังใช้เทคนิค SIMD (Single Instruction Multiple Data) ที่ประมวลผลพิกเซลหลายตัวพร้อมกันในคำสั่งเดียวระดับฮาร์ดแวร์ ต่างจาก loop ของ Python ที่ประมวลผลพิกเซลได้ทีละตัวเท่านั้น ยิ่งไปกว่านั้น box filter ยังสามารถ optimize ด้วยคุณสมบัติ separable filter (แยกคำนวณตามแนวนอนและแนวตั้งได้อิสระ) ทำให้จำนวนการคำนวณลดลงจาก O(K²) เหลือ O(K) ต่อพิกเซล ซึ่งเป็นสิ่งที่โค้ด manual ของผมไม่ได้ทำ
- **หลักฐานจากตัวเลข:** เมื่อขยาย kernel จาก 3×3 เป็น 5×5 (จำนวนพิกเซลต่อหน้าต่างเพิ่มจาก 9 เป็น 25 เท่า) เวลาของ manual filter เพิ่มขึ้นตามสัดส่วน K² อย่างชัดเจน (0.223 → 0.476 วินาที) แต่ `cv2.blur()` แทบไม่ขยับ (0.0043 → 0.0002 วินาที) ยืนยันว่า overhead ที่กล่าวมาข้างต้นสะสมทวีคูณตามจำนวนรอบ loop ใน Python จริง ในขณะที่ optimized C code แทบไม่ได้รับผลกระทบจากขนาด kernel ที่เพิ่มขึ้นในระดับนี้

### 2. การวิเคราะห์การตอบสนองเชิงฟิสิกส์ต่อสัญญาณรบกวน (Noise Filtration Behavior)

**ทำไม Box Filter (เชิงเส้น) เปลี่ยน noise สีขาว (255) ให้กลายเป็นหมอกสีเทา:**
Box Filter คำนวณผลลัพธ์จาก **ผลรวมเชิงเส้น (linear combination)** ของพิกเซลทุกตัวในหน้าต่าง แล้วหารเฉลี่ย นั่นหมายความว่าค่า 255 ที่เป็น noise เดี่ยวๆ จุดหนึ่ง จะถูกนำไป "ผสม" เข้ากับพิกเซลข้างเคียงทุกตัวในหน้าต่างที่มันปรากฏอยู่ ผลคือค่าเฉลี่ยของทุกหน้าต่างที่คาบเกี่ยวจุด noise นั้นจะถูกดึงสูงขึ้นเล็กน้อยเป็นวงกว้าง (ตามขนาด kernel) แทนที่ noise จะหายไป มันกลับ **กระจายพลังงาน (energy) ของตัวเองออกไปสู่พิกเซลรอบข้าง** กลายเป็นรอยเปื้อนสีเทาจางๆ ล้อมรอบตำแหน่งเดิม ยิ่ง kernel ใหญ่ยิ่งกระจายเป็นวงกว้างขึ้นแต่ความเข้มจางลง

**ทำไม Median Filter (ไม่เชิงเส้น) ลบจุด noise ทิ้งได้โดยไม่ทำลายขอบภาพ:**
Median Filter ไม่ได้ใช้ผลรวมเชิงเส้น แต่ **เรียงลำดับ (sort)** ค่าพิกเซลทั้งหมดในหน้าต่างแล้วเลือกค่าที่อยู่ตรงกลาง (median) มาเป็นผลลัพธ์ ด้วยหลักสถิติ ค่าที่เป็น noise แบบ salt-and-pepper (0 หรือ 255) มักเป็นค่าสุดขั้ว (outlier) เพียงจุดเดียวหรือไม่กี่จุดในหน้าต่าง เมื่อเรียงลำดับแล้ว ค่าเหล่านี้จะไปอยู่ที่ปลายสุดของลำดับ (อันดับแรกสุดหรือท้ายสุด) ไม่ใช่ตำแหน่งกลาง จึง **ไม่มีโอกาสถูกเลือกเป็นผลลัพธ์เลย ตราบใดที่พิกเซล noise ไม่เกินครึ่งหนึ่งของหน้าต่าง** นี่คือเหตุผลที่ Median Filter กำจัด noise ได้แบบ "ตัดทิ้งเลย" ไม่ใช่ "เจือจาง" เหมือน Box Filter และเนื่องจากค่าที่เลือกมาเป็นค่าที่ "มีอยู่จริง" ในภาพ (ไม่ใช่ค่าที่คำนวณผสมขึ้นใหม่) บริเวณขอบภาพ (edge) ซึ่งมีการเปลี่ยนแปลงความสว่างอย่างฉับพลันอยู่แล้วโดยธรรมชาติ จึงไม่ถูกทำให้พร่ามัวเหมือนที่ Box Filter ทำ

### 3. การวิเคราะห์พฤติกรรมความผิดเพี้ยนของข้อมูล (The Danger of Overflow)

ถ้าในขั้นตอน Sharpening (โจทย์ข้อที่ 2) ลืมใช้ `np.clip()` และคำนวณในชนิดข้อมูล `np.uint8` โดยตรงตั้งแต่แรก (แทนที่จะคำนวณด้วย float แล้วค่อยแปลงทีหลัง) จะเกิดผลเสียดังนี้:

- **กลไกของ overflow:** ชนิดข้อมูล `uint8` เก็บค่าได้เฉพาะช่วง 0–255 เท่านั้น เมื่อผลการหักลบ (`gray_image - laplacian_gradient`) ได้ค่าที่ติดลบหรือเกิน 255 แต่ถูกบังคับให้อยู่ใน `uint8` ตั้งแต่ต้น ค่าจะไม่ถูก "ตัด" ให้อยู่ในขอบเขต แต่จะเกิดการ **wraparound แบบ modulo 256** แทน เช่น ค่า -10 จะกลายเป็น 246 (คำนวณจาก 256-10) และค่า 300 จะกลายเป็น 44 (คำนวณจาก 300-256)
- **ผลต่อทัศนภาพ:** พิกเซลที่ควรจะมืดสนิท (ค่าติดลบมาก) จะกลับกลายเป็น **จุดสว่างจ้าแบบสุ่ม** และพิกเซลที่ควรจะสว่างจ้า (ค่าเกิน 255 มาก) จะกลับกลายเป็น **จุดมืดสนิทแบบสุ่ม** สลับตำแหน่งกันไปมาแบบไม่มีรูปแบบ (random-looking speckle noise) กระจายอยู่ทั่วภาพโดยเฉพาะบริเวณที่มีค่า gradient สูง
- **ผลต่อความถูกต้องของเส้นขอบ:** เนื่องจากจุดที่เกิด overflow มักอยู่ตรงบริเวณขอบวัตถุ (edge) พอดี (เพราะ Laplacian gradient จะมีค่าสูงสุด/ต่ำสุดตรงขอบ) เส้นขอบที่ควรจะถูกเน้นให้คมชัดขึ้นจากการทำ Sharpening จะกลับกลายเป็น **เส้นประขาด-ดำสลับสีแบบไม่ต่อเนื่อง (broken/speckled edges)** แทนที่จะเป็นเส้นคมชัดต่อเนื่อง ทำลายวัตถุประสงค์หลักของการทำ Sharpening ไปโดยสิ้นเชิง แม้ว่าสูตรทางคณิตศาสตร์ที่ใช้จะถูกต้องทุกประการก็ตาม
- **สรุป:** การใช้ `np.clip(sharpened_image, 0, 255)` ก่อนแปลงเป็น `uint8` จึงไม่ใช่แค่ "ขั้นตอนความสะอาดของโค้ด" แต่เป็น **เงื่อนไขจำเป็นทางวิศวกรรม** เพื่อป้องกันไม่ให้ผลลัพธ์ทางคณิตศาสตร์ที่ถูกต้องถูกทำลายด้วยข้อจำกัดของชนิดข้อมูลตอนแปลงกลับ

จากการทดลองของผมสรุปได้ว่า:
- `cv2.blur()` เร็วกว่ามากเพราะเป็น C/C++ optimized ในขณะที่ `manual_averaging_filter()` ต้องรับ overhead ของ Python
- Box Filter จะทำให้ noise ขาว/ดำกระจายเป็นหมอกสีเทา ส่วน Median Filter จะลบ noise ให้สะอาดกว่าและรักษาขอบได้ดีกว่า
- การทำ Sharpening ต้องจำกัดพิกเซลก่อนแปลงเป็น `uint8` เพื่อหลีกเลี่ยง overflow และรักษาความถูกต้องของเส้นขอบ
